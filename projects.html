My Projects

Building projects is how I learn best, and every application you see here was built to solve a real problem that I or someone I know actually had. I don't believe in building toy apps for the sake of it; I believe in shipping things that work, are deployed to the web, and can be used by real people. Each project on this page is fully deployed and production-ready, meaning it has a live URL, a real database, and handles real user interactions. I have worked with a range of technologies across these projects, including React, Next.js, Flask, PostgreSQL, Supabase, and WebSockets, among others. Some of these projects were built solo, and some were collaborative team efforts, and both have taught me different and equally valuable lessons. I approach every project with the same mindset: start with the user, design the data model, build the backend, then build the frontend on top of it. You can find live links and GitHub repositories for each project below, and I encourage you to explore them and see the work firsthand.

The projects listed here represent a range of different problem domains: sports analytics, productivity tools, entertainment, and health technology. This variety is intentional: I want to be a developer who can tackle problems in any domain, not just one vertical. I believe that strong fundamentals in software engineering transfer across industries, and building in different areas helps me develop that versatility. Each project also pushed me technically in different ways: StatScout taught me API integration and data processing; ApplyTrack taught me authentication flows and state management; BuzzBeats taught me real-time communication with WebSockets; and my capstone taught me how to work with messy, real-world sensor data. I am always working on something new, and this list will continue to grow as I take on more challenges in my final year of school and beyond. If any of these projects interest you or you want to discuss them further, feel free to reach out.

A Note on Technology Choices

I chose React + Flask as my core full-stack combination because React gives me a component-driven frontend with a rich ecosystem, while Flask is lightweight and lets me build clean, well-structured REST APIs in Python. I host frontends on Vercel and backends on Render, with Supabase for managed PostgreSQL databases.

StatScout

React • Flask • PostgreSQL • Vercel • Render • Supabase

StatScout is a full-stack multi-sport analytics platform that gives sports bettors a data-driven edge across NBA player props and soccer match betting. The platform integrates with real-time betting odds APIs to display current sportsbook lines, then computes hit rates and trust scores based on historical performance data to help users identify value bets. The idea came from my own frustration with manually researching player stats before placing bets, and I wanted a tool that could surface the best opportunities automatically. I built the data pipeline in Flask, storing historical game logs and odds data in a PostgreSQL database hosted on Supabase. The React frontend fetches this data and displays it in a clean, filterable dashboard that makes it easy to compare props across different sportsbooks. One of the most technically interesting features is the halftime live projection engine, which takes a player's first-half performance and projects their full-game total using weighted historical averages. StatScout also includes matchup analysis, home/away splits, and opponent defensive rankings to give users context beyond just the raw numbers.

Building StatScout taught me a great deal about third-party API integration, rate limiting, and data normalization across different data sources. I also learned how important caching is: without it, every page load would hit the external odds API and slow everything down significantly. The trust score algorithm was one of the most challenging parts to design, because it needed to balance multiple factors like hit rate, sample size, and recent form without overweighting any single variable. I iterated on that algorithm several times before arriving at a formula I was satisfied with, and I documented the logic in the codebase so future contributors can understand and improve it. The project is live at statscout.vercel.app and I continue to expand sport coverage and add new features.

Key Features

  • Real-time betting odds across NBA and soccer via third-party API integration
  • Hit rate and trust score calculations with custom algorithms
  • Halftime live projection engine
  • Matchup history and opponent defensive rankings
  • Dark mode, responsive design, and persistent user preferences
  • Parlay builder for combining multiple props

ApplyTrack

React • Flask • PostgreSQL • JWT • Vercel • Render

ApplyTrack is a full-stack job application management platform built to help job seekers (including myself) stay organized during a job search. The idea was simple: I was applying to dozens of positions and losing track of where each one stood in the process, so I built a tool to solve that exact problem. Users can create a secure account, log in with JWT-based authentication, and track every application from submission through offer in one centralized place. The platform offers two views: a traditional list view with search and filter capabilities, and a Kanban board where users can drag applications through stages like Applied, Phone Screen, Interview, Offer, and Rejected. A dashboard provides summary statistics (total applications, response rate, interviews scheduled) so users can measure their job search performance at a glance. I also implemented a password reset via email feature, which required setting up an email service on the backend and securely generating and validating time-limited reset tokens. The entire application has dark mode support, with the user's preference stored in localStorage so it persists between sessions.

This project pushed my skills in authentication and session management significantly, since handling JWTs securely (including refresh tokens, expiration, and logout) requires careful design. I also learned a lot about state management in React by building the Kanban board, which requires complex, nested state updates when cards move between columns. The dashboard charts are built with Recharts, a React charting library, which was new to me and required diving into documentation to customize the visualizations the way I wanted them. I focused heavily on the user experience of this app: every interaction, from adding an application to updating its status, should feel fast and frictionless. ApplyTrack is live at job-tracker-three-ochre.vercel.app and is the project I most often reference in interviews.

Key Features

  • JWT-based authentication with password reset via email
  • Kanban board with drag-and-drop stage management
  • Dashboard with application statistics and charts
  • Full CRUD operations for job application records
  • Dark mode toggle with local storage persistence

BuzzBeats

Next.js • Node.js • WebSockets • Spotify API

BuzzBeats is a real-time multiplayer music quiz game built for people who love music and a bit of friendly competition. The concept is straightforward: a host creates a game lobby, selects a playlist or genre, and players join via a shared code on their own device. Once the game starts, audio clips are played and players race to identify the song or artist as quickly as possible, as both speed and accuracy affect your score. The real-time aspect is powered by WebSockets, which allow the host and all connected players to stay in sync without any page refreshes or polling. This was my first time building a project with WebSockets, and it was one of the most challenging and rewarding technical experiences I have had. Designing the event model (what messages get sent, when, and to whom) required careful thinking about race conditions, disconnections, and state synchronization across clients. The project was built as a team effort, which also gave me experience with collaborative Git workflows, code reviews, and dividing responsibilities across a shared codebase.

BuzzBeats is currently in active development and has not yet been deployed to a public URL, but the core gameplay loop is fully functional locally. We are working on adding a Spotify integration to allow hosts to pull from their own playlists, which requires handling OAuth authentication with the Spotify API. The frontend is built in Next.js, which gave us server-side rendering capabilities and a clean routing structure for the different game views: the host lobby, player lobby, game screen, and results screen. Future plans include leaderboards, custom round timers, and the ability to save and replay past games. Once deployed, this will be the project I am most excited to share because it is genuinely fun to play with friends.

Key Features

  • Real-time multiplayer via WebSocket connections
  • Host and player roles with separate control panels
  • Live leaderboard and per-round scoring
  • Lobby system with shareable join codes
  • Music integration for audio clip playback during rounds

Multi-Sensor Data Visualization Toolkit

Python • Dash • Plotly • Machine Learning • GitLab

This is my year-long senior capstone project, developed in collaboration with a research partner and a team of four engineers. The toolkit analyzes multi-sensor time-series data collected from health monitoring wearables, applies machine learning models for anomaly detection, and presents findings through an interactive web dashboard built with Python Dash and Plotly. The research partner provided real sensor datasets from ongoing health studies, which meant we were working with messy, real-world data (missing values, sensor drift, and inconsistent sampling rates) rather than clean textbook datasets. Building the data ingestion and preprocessing pipeline to handle this variability was one of the most complex engineering challenges I have faced. Our anomaly detection model uses a combination of statistical thresholding and a trained isolation forest to identify patterns in the data that may indicate health events worth flagging for researchers. The visualization dashboard is designed to be accessible to non-technical health researchers, so we put significant effort into making the charts informative and the interface intuitive, even for users with no data science background. The project is developed using Agile methodology, with two-week sprints, regular standups, and a GitLab CI/CD pipeline for automated testing and deployment.

Key Features

  • ML-based anomaly detection on time-series sensor data
  • Interactive data visualization dashboard with Plotly and Dash
  • Data processing pipelines for multi-sensor input
  • Agile development with sprint planning and GitLab CI/CD
